⚠  NetworkPolicy enforcement preflight (NET-07):
  The isolation invariant above is enforced by this chart's NetworkPolicies.
  Those objects render correctly on ANY cluster, but they only take EFFECT if
  your CNI actually enforces NetworkPolicies. A non-enforcing CNI (e.g. stock
  `kindnet`) leaves Sandbox B → Sandbox A reachable with no error at install
  time — the air-gap is silently inert.
  • A NetworkPolicy-enforcing CNI (Calico or Cilium) is a HARD prerequisite.
    See docs/CUSTOMER_HELM_RUNBOOK.md#prereqs-1-time for the proven remedies
    (Calico v3.27.3 / GKE Dataplane V2 / Azure or Calico network policy).
  • To actively confirm your CNI enforces NetworkPolicies, run:
      kubectl apply -f docs/cni-probe-job.yaml
      kubectl wait --for=condition=complete -n lucairn-probe job/lucairn-cni-probe --timeout=90s
      kubectl logs -n lucairn-probe job/lucairn-cni-probe
    PASS — blocked flow rejected; CNI enforcing NetworkPolicy.
    FAIL — blocked flow reachable; CNI NOT enforcing; isolation air-gap is inert.
    Clean up: kubectl delete namespace lucairn-probe
    (`lucairn doctor` only checks that NetworkPolicy OBJECTS are present, not
    that they are enforced.)

Gateway keystore encryption (#17):
  GATEWAY_KEYSTORE_KEY is REQUIRED for gateway boot. The chart wires it via
  the gateway-credentials Secret. Provision per backend:
    k8s-native       — pass via: --set gateway.secrets.values.gatewayKeystoreKey="$(openssl rand -base64 32)"
    vault/aws/azure  — populate property `gatewayKeystoreKey` in the remote store
  Gateway will refuse to boot without it (fail-closed). Format: base64 of 32
  raw bytes (AES-256). Rotation requires re-keying the on-disk keystore;
  re-running install with a different value invalidates every encrypted record.

=======================================================
  ⚠  DATA DESTRUCTION ON UNINSTALL — READ BEFORE `helm uninstall`
=======================================================

The compliance databases (audit, id-bridge, veil-witness) hold your
REGULATOR-CITED EVIDENCE: the immutable audit trail (AI Act Art. 12 /
GDPR Art. 30), the re-linkage tokens (GDPR DSAR / erasure), and the
signed witness certificates. This data is NOT recoverable once gone.

  • The PVCs for these three databases carry `helm.sh/resource-policy: keep`,
    so `helm uninstall` does NOT delete them. The data SURVIVES an uninstall
    and is re-bound on the next install into the same namespaces.
  • To DELETE the data on purpose you must remove the PVCs explicitly:
      kubectl delete pvc audit-postgresql-data       -n dsa-audit
      kubectl delete pvc id-bridge-postgresql-data    -n dsa-bridge
      kubectl delete pvc veil-witness-postgresql-data  -n dsa-witness
  • BEFORE any uninstall/reinstall or PVC deletion: take a backup.
    See OPS.md § Backups for the pg_dump / pg_restore recipes (Compose + Helm)
    and the restore-validation step.
